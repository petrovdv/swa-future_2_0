# Стратегический роадмап внедрения Data Mesh

## 1. Ключевые роли и зоны ответственности
| Роль                              | Ответственность                                                                                                                                              | Взаимодействие с доменами                      |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------- |
| **Data Product Owner (DPO)**      | Владелец данных в домене (Клиники, Финтех, ИИ, Партнёры). Определяет SLA, качество, контракт доступа, отвечает за соответствие 152-ФЗ внутри своего контура. | Бизнес-подразделения, BI-аналитики, Compliance |
| **Data Engineer**                 | Проектирование и поддержка пайплайнов: Debezium CDC → Kafka → Flink/Spark → ClickHouse Read Models. Оптимизация вычислений в РФ-облаках.                     | Платформенная команда, DPO                     |
| **BI / Analytics Engineer**       | Настройка Self-Service BI (Yandex DataLens / Superset / Visiology), семантических слоёв, RLS/ABAC, обучение бизнес-пользователей.                            | Бизнес-пользователи, DPO                       |
| **Platform / Data Architect**     | Стандарты событий, Apicurio Schema Registry, каталог данных (DataHub), инфраструктура IaC (Yandex Cloud/Selectel), управление кластерами.                    | Все домены, DevOps, Compliance                 |
| **Compliance & Security Officer** | Контроль изоляции ПДн/медицинских данных, аудит доступов, интеграция СКЗИ (КриптоПро/VipNet), соответствие 152-ФЗ, ЦБ РФ, требованиям ФСТЭК.                 | Юридический отдел, DPO, Platform Architect     |

## 2. Фазы внедрения (0–36 месяцев)

### Фаза 1: Пилот и закладка фундамента (0–6 месяцев)
| Этап | Действия | Роли | Бизнес-цель |
|------|----------|------|-------------|
| **Пилот домена** | Выбор Финтех (платежи/кредиты). Развёртывание Yandex Managed Kafka / self-hosted Kafka, Apicurio Schema Registry, Debezium CDC из SQL 2008. Настройка контуров шифрования. | Platform Architect, Data Engineer, Compliance | Снижение time-to-market для новых финансовых продуктов, запуск событийной шины в РФ-облаке |
| **Первые Data Products** | Формирование витрин `CashFlow`, `CreditRisk` в Yandex ClickHouse. Настройка RBAC/ABAC и Row-Level Security в DataLens/Superset. | DPO (Финтех), BI Analyst | Запуск self-service отчётности для риск-менеджеров без выгрузки сырых данных |
| **Стандарты и Governance** | Утверждение контрактов событий, DLQ-стратегии, политик ретайна. Интеграция ГОСТ-шифрования для передачи ПДн. Аудит соответствия 152-ФЗ и ЦБ РФ. | Platform Architect, Compliance, Security | Подготовка к масштабированию без деградации качества и рисков регуляторных штрафов |

### Фаза 2: Масштабирование и потоковая аналитика (6–18 месяцев)
| Этап | Действия | Роли | Бизнес-цель |
|------|----------|------|-------------|
| **Расширение доменов** | Подключение Клиник (`Visit`, `Patient`) и ИИ-сервисов (`DiagnosticTask`). Настройка ACL для легаси. Сегментация контуров: ПДн/медицинские данные изолированы от финансовых потоков. | Data Engineer, DPO (Клиники/ИИ), Compliance | Независимое развитие направлений без блокировок DWH, соблюдение локализации данных |
| **Stream Read Models** | Переход от batch-ETL к Apache Flink / Kafka Streams. Материализация витрин в near-real-time в ClickHouse. Оптимизация ресурсов через авто-масштабирование в Selectel/VK Cloud. | Data Engineer, BI Analyst, Platform Architect | Отчёты формируются за секунды, а не часы. Снижение нагрузки на production-БД |
| **Data Catalog & Discovery** | Запуск self-hosted DataHub/Amundsen. Автоматический lineage, поиск витрин бизнес-пользователями, каталогизация контрактов событий. | Platform Architect, BI Analyst, DPO | Сокращение времени на поиск и валидацию данных на ~50%, прозрачность источников |
| **Мультимегиональность в РФ** | Шаблонизация развёртывания (Terraform + Yandex Cloud/Selectel providers). Подготовка к репликации шины в 2–3 региона РФ (Москва, СПб, Екатеринбург/Новосибирск) с соблюдением 152-ФЗ. | Platform Architect, DevOps, DPO | Готовность к выходу в новые регионы РФ с локальным хранением ПДн и низкой задержкой |

### Фаза 3: Зрелая Data Mesh и near-real-time (18–36 месяцев)
| Этап | Действия | Роли | Бизнес-цель |
|------|----------|------|-------------|
| **Отказ от синхронных интеграций** | Вывод Apache Camel ESB и SQL 2008 DWH из критического пути. Полное взаимодействие через события. Легаси остаётся только как исторический архив. | Platform Architect, Data Engineer | Устранение единой точки отказа, снижение TCO на 15–20%, упрощение поддержки |
| **Монетизация ИИ-функций** | Публикация событий `DiagnosticCompleted` для внешних партнёров (фарма/медтех) через Yandex API Gateway / self-hosted Kong. Контракты с ГОСТ-шифрованием и аудитом доступа. | DPO (ИИ), Compliance, Security | Новые revenue streams, безопасный обмен данными с внешними экосистемами |
| **Автоматизация DataOps** | CI/CD для витрин и пайплайнов (GitLab CI / Yandex Cloud CI), тестирование контрактов событий, auto-scaling кластеров, FinOps-мониторинг рублёвых затрат. | Data Engineer, Platform Architect, FinOps | Поддержка сотен ТБ данных без ручного вмешательства, прозрачная оптимизация облачных расходов |
| **Оптимизация затрат и ресурсов** | Переход на preemptible/spot-инстансы для batch-агрегаций, tiering хранения (Yandex Object Storage: Standard → Cold/Archive), долгосрочные контракты с провайдерами. | FinOps Analyst, DPO, Platform Architect | Снижение операционных расходов на 15–20% без потери SLA 99.95% |

## 3. Метрики успеха (KPI)
| Направление | Метрика | Целевое значение (36 мес) |
|-------------|---------|---------------------------|
| **Time-to-Market** | Среднее время вывода нового отчёта / витрины | ≤ 1 рабочего дня |
| **Качество данных** | % событий с нарушенным контрактом или DLQ-ошибками | < 0.1% |
| **Доступность** | SLA витрины и шины событий (по регионам РФ) | ≥ 99.95% |
| **Самообслуживание** | Доля отчётов, созданных бизнес-аналитиками без привлечения Data Engineers | ≥ 70% |
| **Стоимость (FinOps)** | TCO на ТБ обработанных данных (в руб.) | -25% от базового уровня (Год 0) |
| **Комплаенс** | Прохождение аудитов 152-ФЗ / ЦБ РФ без критических замечаний | 100% (ежегодно) |

## 4. Управление рисками (РФ-специфика)
| Риск | Митигация |
|------|-----------|
| **Сопротивление переходу на Self-Service** | Обучение бизнес-аналитиков работе с DataLens/Superset, назначение BI Champions в доменах, геймификация каталога данных |
| **Нарушение обратной совместимости контрактов** | Mandatory Apicurio Schema Registry, breaking-change reviews, канареечные потребители, автоматические тесты в CI/CD |
| **Перегрузка шины в часы пик** | Партиционирование по `tenantId`/`domain`, backpressure, авто-масштабирование Kafka/Flink в облаке, приоритизация топиков |
| **Утечка ПДн/медицинских данных через витрину** | Строгий ACL на уровне подписки, автоматическая маскировка полей, обязательное ГОСТ-шифрование (СКЗИ), аудит доступа через KMS/Lockbox |
| **Дефицит кадров Data Engineering в РФ** | Инвестиции во внутреннюю академии, партнёрство с вузами, автоматизация пайплайнов, использование low-code инструментов (DataLens), переход на open-source стек с коммерческой поддержкой |
| **Зависимость от одного облачного провайдера** | Использование Terraform/IaC для абстракции инфраструктуры, мульти-облачная стратегия (Yandex Cloud + Selectel/VK Cloud), регулярные DR-тесты |

> **Принцип внедрения:** Data Mesh в российских реалиях — это не только технологическая трансформация, но и организационная перестройка с обязательным учётом регуляторных требований (152-ФЗ, ЦБ РФ, ГОСТ/СКЗИ). Технологии (Kafka, ClickHouse, DataLens, DataHub) выступают инфраструктурным фундаментом. Успех зависит от назначения ответственных Data Product Owners, чётких SLA, культуры владения данными внутри доменов и непрерывного комплаенс-контроля.
