
# Обоснование перехода к событийной архитектуре

### 1. Ограничения текущей архитектуры (DWH + Camel ESB)

- **Жёсткая связность:** Бизнес-логика реализована в хранимых процедурах SQL Server 2008. Изменение одного отчёта требует пересборки ETL-пайплайнов и координации нескольких команд.
- **Пакетная обработка:** DWH обновляется ночными батчами. Аналитика отстает от реальности на 12–24 часа, что критично для финтеха и оперативного управления клиниками.
- **Единая точка отказа:** Рост данных до сотен ТБ приводит к блокировкам при длительных трансформациях. Camel ESB становится bottleneck при синхронных интеграциях.
- **Сложность масштабирования:** Добавление нового направления (фарма, новые регионы) требует модификации централизованной схемы DWH, что замедляет time-to-market.

### 2. Преимущества Event-Driven Architecture (EDA)

| Критерий                   | DWH + Camel (текущее)                                    | EDA (целевое)                                                                     |
| -------------------------- | -------------------------------------------------------- | --------------------------------------------------------------------------------- |
| **Гибкость**               | Изменения требуют перестройки ETL и согласований         | Новые подписчики подключаются без изменения источников (publish/subscribe)        |
| **Масштабируемость**       | Вертикальное/ограниченное горизонтальное масштабирование | Независимое масштабирование доменов; партиционирование потоков данных             |
| **Скорость реакции**       | Отчёты формируются часами (batch)                        | Near-real-time через материализованные представления (CQRS + stream processing)   |
| **Отказоустойчивость**     | Каскадные блокировки при падении DWH                     | Асинхронные очереди, DLQ, retry-механизмы; изоляция сбоев по доменам              |
| **Безопасность/Комплаенс** | Прямой доступ к операционным данным из BI                | Витрина потребляет только агрегированные события; мед. данные исключены из потока |

### 3. Влияние на бизнес-цели «Будущее 2.0»

- **Time-to-market:** Новые продукты запускаются как независимые подписчики на существующие события. Не требуется рефакторинг DWH.
- **Геомасштабирование:** Региональные узлы разворачиваются локально и синхронизируются через глобальную шину событий, соблюдая требования локальных регуляторов.
- **Переход к near-real-time:** Потоковая обработка (Kafka + Spark Streaming) заменяет пакетные ETL. Сложные отчёты собираются из pre-aggregated read models за секунды.
- **Безопасность данных:** Разделение write/read моделей и фильтрация событий на уровне политик гарантируют, что медицинские карты и истории болезней остаются в изолированном контуре, соответствуя требованиям регуляторов.
