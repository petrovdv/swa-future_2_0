# Terraform CI/CD: Yandex Cloud VM Module

## Описание

Автоматизированное развёртывание виртуальных машин в Yandex Cloud через GitHub Actions.  
Состояние инфраструктуры хранится удалённо в **Yandex Object Storage**, аутентификация происходит через **Service Account JSON Key**.  
Конфигурация модульная, поддерживает несколько окружений (`dev`/`stage`/`prod`) и изолированные state-файлы для Pull Request'ов.

## Принцип работы пайплайна

| Триггер                                | Job         | Условие                         | Действие                                | State-файл                      |
| -------------------------------------- | ----------- | ------------------------------- | --------------------------------------- | ------------------------------- |
| `pull_request`                         | `Plan (PR)` | Изменения в `*.tf` / `*.tfvars` | Автогенерация плана, **без применения** | `pr-<NUMBER>/terraform.tfstate` |
| `workflow_dispatch` + `action=deploy`  | `Deploy`    | Ручной запуск, выбор окружения  | План + автоматическое применение        | `<env>/terraform.tfstate`       |
| `workflow_dispatch` + `action=destroy` | `Destroy`   | Ручной запуск + подтверждение   | План уничтожения + выполнение           | `<env>/terraform.tfstate`       |

### Безопасность

- Все секреты хранятся в **GitHub Secrets**, маскируются в логах
- State-файлы в **Yandex Object Storage** с включённым `Versioning` (история изменений)
- Аутентификация через `SA_KEY_JSON` → IAM-токен обновляется автоматически
- Конкурентный доступ: `concurrency` блокирует параллельные запуски на одно окружение
- `confirm_destroy` — защита от случайного удаления: требуется явное подтверждение

## Подготовка к запуску

### 1. Yandex Cloud

1. Создайте бакет, например `tf-state-mydev` → включите **Versioning**
2. Создайте сервисный аккаунт `terraform-ci` → назначьте роли:
    - `compute.editor`, `vpc.viewer` — на целевой каталог
    - `storage.editor` — на бакет с состоянием
    - TODO: при тестировании оказалось, что указанного набора ролей недостаточно, необходимо определить минимальный набор ролей, 
    - работа скриптов была проверена с полными правами (роль admin)
3. Сгенерируйте **два ключа**:
    - **Статический S3-ключ** → для доступа к бакету: `S3_ACCESS_KEY`, `S3_SECRET_KEY`
    - **IAM JSON-ключ** → для аутентификации в облаке: `SA_KEY_JSON` (многострочный)

### 2. GitHub Secrets

Добавьте в `Settings → Secrets and variables → Actions`:

|Secret|Пример значения|Описание|
|---|---|---|
|`YC_CLOUD_ID`|`b1gxxxxxxxxxxxx`|ID облака Yandex Cloud|
|`YC_FOLDER_ID`|`b1exxxxxxxxxxxx`|ID каталога для ресурсов|
|`STATE_BUCKET`|`tf-state-mydev`|Имя бакета для state-файлов|
|`S3_ACCESS_KEY`|`YCAXXXXXXXXXXXXX`|`key_id` из статического S3-ключа|
|`S3_SECRET_KEY`|`XXXXXXXXXXXXXXXXXXXX`|`secret` из статического S3-ключа|
|`SA_KEY_JSON`|`{ "id": "...", "private_key": "..." }`|Полное содержимое IAM-ключа (в одну строку или с переносами)|

### 3. GitHub Environments

Для защиты окружений настройте approval-гейты:

| Environment | Required reviewers | Purpose                |
| ----------- | ------------------ | ---------------------- |
| `dev`       | (опционально)      | Быстрое тестирование   |
| `stage`     | команда QA         | Предпродакшн-валидация |
| `prod`      | тимлид + DevOps    | Продакшн-релиз         |

> Джобы `Deploy` и `Destroy` будут ждать ручного подтверждения, если включены ревьюверы.

##  Использование

###  Просмотр плана изменений (автоматически)

1. Создайте или обновите Pull Request с изменениями `.tf` / `.tfvars`
2. Дождитесь завершения проверки **`Plan (PR)`** во вкладке **Checks**
3. Результат `terraform plan` будет прикреплён к комменту в PR

###  Ручной деплой

1. Перейдите: `Actions → Terraform CI/CD → Run workflow`
2. Заполните параметры
3. Нажмите **Run workflow**
4. Если для `stage` настроены ревьюверы — дождитесь подтверждения
5. После успешного завершения ресурсы появятся в облаке

### Уничтожение окружения

1. `Actions → Terraform CI/CD → Run workflow`
2. Заполните параметры
3. Нажмите **Run workflow**
4. Проверьте логи: сначала `Destroy Plan`, затем `Execute Destroy`
